# 🪕 우쿨렐레 코드 운지 교정 웹앱 — 바이브코딩 프롬프트 설계서

> **목표:** 웹캠으로 운지하는 손을 촬영해, 지금 잡은 코드가 목표 코드와 맞는지 판정하고,
> 틀렸다면 **"몇 번 손가락을 어느 줄·프렛으로 옮겨라"** 까지 구체적으로 교정해 주는 웹앱.
>
> 이 문서는 그대로 읽는 설명서가 아니라, **AI 코딩 도구(Claude Code / Cursor / Codex / Copilot 등)에 단계별로 복붙해서
> 점진적으로 완성**하기 위한 "프롬프트 모음 + 설계 근거"입니다.
>
> 📌 이 설계서는 검증된 **기타 코드 운지 교정 웹앱** 설계서를 우쿨렐레에 맞게 다시 설계한 것입니다.
> 파이프라인은 같지만, **줄 4개·좁은 프렛·re-entrant 튜닝·1~2손가락 코드** 같은 우쿨렐레 고유 특성을 전부 반영했습니다.

---

## 0. 먼저 알아야 할 핵심 진실 (이걸 모르면 헤맵니다)

이 기능의 진짜 난이도는 "손 인식"이 아니라 **"손가락 끝이 지판 위 어느 줄·어느 프렛에 있는지 아는 것"** 입니다.
순진하게 "AI한테 코드 잡은 거 맞는지 봐줘"라고 하면 절대 안정적으로 안 됩니다. 이유:

1. **손 추적은 이미 풀린 문제다.** → MediaPipe Hand Landmarker가 손가락 21개 관절 좌표를 실시간으로 줍니다.
2. **진짜 어려운 건 "지판 좌표계"다.** → 카메라에 비친 픽셀 좌표를 "2번 줄 1번 프렛" 같은 **지판 격자 좌표**로
   바꿔야 합니다. 이건 컴퓨터비전(원근 변환 = homography) 문제입니다.
3. **자동 지판 인식은 깨지기 쉽다.** → 조명·배경·악기 색에 따라 들쭉날쭉. **그래서 MVP는 "수동 캘리브레이션"으로 시작**합니다.
   (사용자가 화면에서 너트·프렛·줄 모서리를 클릭 → 그 4점으로 지판 격자를 만든다.)
4. **카메라 각도가 성패를 가른다.** → 연주 자세 그대로면 손등이 카메라를 가려 손끝이 안 보입니다.
   **지판을 정면(관객 시점)에서 비추는 셋업**을 전제로 설계합니다. (아래 1-3장)

### 🪕 우쿨렐레라서 특별히 신경 써야 할 것 (기타와 다른 점)

| 항목 | 기타 | 우쿨렐레 | 설계 영향 |
|---|---|---|---|
| 줄 개수 | 6개 | **4개 (G·C·E·A)** | 좌표계 `v∈[0,3]`, `string=4-round(v)`, 캘리브 대응점 변경 |
| 줄 번호/위치 | 6번=맨 위(굵음) | **4번(G)=맨 위, 1번(A)=맨 아래** | 캘리브레이션 클릭 순서·교정 문구가 4번줄 기준 |
| 튜닝 | 저음→고음 | **re-entrant(4번 G가 high-G, 얇음)** | "굵은 줄/얇은 줄" 표현 금지 → **줄 번호 기준**으로 교정 |
| 프렛 간격 | 넓음(~36mm) | **좁음(~25mm), 넥도 짧음** | 손끝 떨림·프렛 오판정 ↑ → 스무딩/홀드 게이트가 더 중요 |
| 운지 복잡도 | 3~5손가락 | **1~2손가락 코드 많음 (C·Am은 1개)** | MVP 검증이 쉬움 → 첫 테스트는 C·Am로 |
| 뮤트(x) | 자주 있음 | **거의 없음(보통 4줄 다 침)** | 데이터 모델 단순화, 뮤트 검사는 옵션으로만 |

> 💡 **설계 철학:** "완벽한 자동 인식"을 처음부터 노리지 말 것. **수동 캘리브레이션 + 손 랜드마크 + 격자 매핑**으로
> 동작하는 MVP를 먼저 만들고, 그 다음에 자동화·정밀도를 올린다. 단계마다 "눈으로 확인 가능한 결과물"이 나오게 쪼갠다.
>
> 🎯 **우쿨렐레의 이점:** C(1번줄 3프렛 한 손가락)·Am(4번줄 2프렛 한 손가락)처럼 **단일 손가락 코드**가 있어,
> 손끝→프렛 매핑이 맞는지 가장 빨리 검증할 수 있습니다. Phase 3~4 첫 테스트는 반드시 C·Am로 하세요.

## 1. 기술 스택 & 셋업 권장안

### 1-1. 권장 스택
| 영역 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | **바닐라 JS + HTML/CSS** (또는 Vite) | 라이브러리 가볍게, 카메라/캔버스 직접 제어 |
| 손 추적 | **MediaPipe Tasks Vision — Hand Landmarker** | 브라우저에서 실시간, 손 2개·관절 21개·왼/오른손 구분 |
| 그리기 | **Canvas 2D** (비디오 위 오버레이) | 손끝·지판 격자·교정 화살표를 겹쳐 그림 |
| 좌표 변환 | **4점 Homography (DLT)** 직접 구현 | 외부 의존 최소화. (정밀도 필요 시 OpenCV.js로 교체 가능) |
| 음성 피드백 | **Web Speech API (SpeechSynthesis)** | "약지를 한 칸 위로" 같은 음성 코칭 |

> 대안: React를 써도 무방하나, **실시간 캔버스 렌더 루프**가 핵심이라 상태관리는 가볍게 유지.
> OpenCV.js는 자동 지판 인식(Phase 6)에서만 도입 권장 — MVP엔 과함.

### 1-2. MediaPipe 손가락 끝 랜드마크 (외워둘 것)
```
엄지 끝 = 4 | 검지 끝 = 8 | 중지 끝 = 12 | 약지 끝 = 16 | 새끼 끝 = 20
손목 = 0  | 각 손가락은 뿌리→끝 순서로 4개 관절
```
우쿨렐레 운지 손가락 번호 매핑: **검지=1, 중지=2, 약지=3, 새끼=4, 엄지=T**

### 1-3. 우쿨렐레 줄 번호 (반드시 고정해서 외울 것)
```
4번줄 = G (맨 위, 연주자 얼굴 쪽)   ← 캘리브레이션 v=0
3번줄 = C (가장 굵은 줄)
2번줄 = E
1번줄 = A (맨 아래, 가장 얇은 줄)   ← 캘리브레이션 v=3
```
> ⚠️ **re-entrant 튜닝 주의:** 표준 우쿨렐레는 4번줄(G)이 high-G로, 두께가 가장 얇습니다(3번 C가 가장 굵음).
> 그래서 **"굵은 줄/얇은 줄"이라는 표현은 혼란을 줍니다.** 교정 문구는 항상 **"N번줄"** 번호로만 말하세요.
> 단, "위치"는 일정합니다 — 4번줄이 맨 위, 1번줄이 맨 아래. 운지 판정은 **위치만** 쓰므로 튜닝은 영향 없습니다.

### 1-4. 카메라 셋업 가이드 (앱 안내문에도 넣을 것)
- 📷 카메라를 **지판 정면**(관객이 보는 방향)에 둔다. 폰 거치대/삼각대 권장.
- 💡 지판이 밝고, 손과 배경의 대비가 크도록 조명.
- 🪕 우쿨렐레는 스탠드에 세우거나 무릎 위에서 **지판이 카메라를 향하게** 살짝 눕힌다.
- 🔎 **우쿨렐레는 작다.** 지판이 화면에서 충분히 크게(가로폭의 50% 이상) 잡히도록 카메라를 가까이 둔다. 프렛 간격이 좁아 멀면 오판정↑.
- ✋ 손가락 끝(누르는 지점)이 카메라에서 보여야 한다. 손등으로 가리지 않게.

## 3. 단계별 개발 프롬프트 (Phase 0 → 6)

> 각 Phase는 **① 목표 ② AI에 줄 프롬프트(복붙용) ③ 완료 기준 ④ 흔한 함정** 으로 구성.
> 반드시 **한 Phase씩** 진행하고, 완료 기준을 눈으로 확인한 뒤 다음으로 넘어가세요.

---

### ▶ Phase 0 — 프로젝트 뼈대 + 웹캠 띄우기
**목표:** 카메라 권한 받고 비디오를 화면에 띄운다. (가장 단순한 동작 확인)

> **[복붙 프롬프트]**
> "바닐라 JS + HTML 단일 페이지로 시작하는 웹앱을 만들어줘.
> - `index.html`, `style.css`, `main.js` 3파일.
> - 화면 중앙에 `<video>`(웹캠 미러링, `transform: scaleX(-1)`)와 그 위에 정확히 겹치는 `<canvas>`를 둔다.
> - '카메라 시작' 버튼을 누르면 `navigator.mediaDevices.getUserMedia({video:{width:1280,height:720}})`로 스트림을 받아 비디오에 연결.
> - 권한 거부/미지원 시 사용자에게 친절한 한국어 에러 메시지 표시.
> - 비디오와 캔버스의 픽셀 크기를 항상 일치시키는 리사이즈 핸들러 포함.
> - 다크 테마의 깔끔한 UI. 상단에 앱 제목 '우쿨렐레 코드 운지 교정기'."

**완료 기준:** 버튼 클릭 → 내 모습이 거울처럼 보임. 캔버스가 비디오 위에 정확히 겹침.
**함정:** video와 canvas의 CSS 크기와 내부 해상도(width/height 속성)를 혼동하면 좌표가 어긋남. 둘을 항상 동기화.

---

### ▶ Phase 1 — MediaPipe 손 랜드마크 실시간 표시
**목표:** 손 21개 관절을 캔버스에 점·선으로 그린다. 왼/오른손 라벨 표시.

> **[복붙 프롬프트]**
> "MediaPipe Tasks Vision의 **HandLandmarker**를 CDN(`@mediapipe/tasks-vision`)으로 추가해줘.
> - `runningMode: 'VIDEO'`, `numHands: 2`로 초기화. WASM/모델은 공식 CDN URL 사용.
> - `requestAnimationFrame` 루프에서 `detectForVideo()`를 호출하고, 결과 랜드마크를 캔버스에 그린다:
>   관절은 점, 손가락 뼈대는 선으로(좌우손 색 구분). 각 손끝(4,8,12,16,20)은 더 크게.
> - 손 옆에 `handedness`(Left/Right)와 신뢰도를 텍스트로 표시.
> - 좌표는 정규화값(0~1)이므로 `x*canvas.width`, `y*canvas.height`로 변환. **미러링된 비디오와 좌표계가 맞는지** 확인(필요 시 x를 1-x로).
> - FPS를 화면 구석에 표시."

**완료 기준:** 손을 움직이면 골격이 실시간으로 따라옴. 손끝 점이 실제 손끝과 일치.
**함정:** **미러링**. 비디오를 `scaleX(-1)`로 뒤집었다면 랜드마크 x도 뒤집어야 시각적으로 맞음. 여기서 안 맞추면 이후 전부 어긋남. → 미러 여부를 한 변수로 통일 관리.

---

### ▶ Phase 2 — 지판 캘리브레이션 (이 앱의 심장 ❤️)
**목표:** 사용자가 화면에서 지판 모서리를 클릭 → 원근 변환(homography)으로 **"지판 격자 좌표계"**를 만든다.

**개념:** 너트(0번 프렛)부터 기준 프렛 K까지, **4번줄(G)~1번줄(A)**을 덮는 **사각형의 4모서리**를 클릭받아,
이미지 픽셀 → 지판좌표 `(u, v)`로 보내는 3×3 변환행렬 H를 구한다.
- `u`: 너트=0, 프렛K=K (네크 방향, 프렛 인덱스)
- `v`: **4번줄=0, 1번줄=3 (줄 방향, 4줄)**

> **[복붙 프롬프트]**
> "지판 캘리브레이션 기능을 추가해줘. **우쿨렐레는 줄이 4개(G·C·E·A)다.**
> 1. '지판 보정' 버튼 → 안내: '4번줄(G)·너트' → '1번줄(A)·너트' → '1번줄(A)·K번 프렛' → '4번줄(G)·K번 프렛' 순서로 4점을 클릭하라고
>    화면에 오버레이로 단계별 안내. (K는 입력값, **기본 5**, 최소 3, 최대 12)
> 2. 클릭한 4점(이미지 픽셀)을 각각 지판좌표 `(0,0),(0,3),(K,3),(K,0)`에 대응시키는 **homography 행렬 H**(이미지→지판)와
>    역행렬 H⁻¹(지판→이미지)를 4점 DLT로 계산하는 순수 함수 `computeHomography(srcPts, dstPts)`를 직접 구현.
> 3. 검증용 오버레이: H⁻¹로 지판 격자(프렛선 0~K, **줄 4개**)를 다시 이미지에 투영해 **녹색 격자**로 그린다.
>    격자가 실제 지판과 잘 겹치면 보정 성공.
> 4. 보정값은 localStorage에 저장하고 다음 실행에 복원.
> ⚠️ 실제 프렛 간격은 균등하지 않다(고음으로 갈수록 좁아짐). 우쿨렐레는 넥이 짧아 더 촘촘하다.
>    내부 프렛선을 그릴 땐 평균율 공식 `프렛n 위치비율 = 1 - 2^(-n/12)`로 보정해 격자선을 배치해줘."

**완료 기준:** 4점 클릭 후 녹색 지판 격자가 실제 우쿨렐레 지판 위에 거의 정확히 겹친다(줄 4개·프렛선 K+1개).
**함정:**
- 4점 순서를 섞으면 격자가 뒤집힘 → 안내 순서를 화면에 명확히.
- 카메라/악기가 움직이면 보정이 깨짐 → "재보정" 버튼 항상 노출. 셋업 후 고정 권장.
- **우쿨렐레는 작아서 클릭 정밀도가 부족하면 격자 오차가 크다** → 지판을 화면에서 크게 잡고, 확대 클릭(돋보기) 옵션 고려.
- 선형 homography만 쓰면 고프렛에서 오차 큼 → 위 평균율 공식으로 프렛선 위치 보정.

### ▶ Phase 3 — 손끝 → 지판 격자 좌표 매핑
**목표:** 각 손가락 끝 픽셀좌표를 H로 변환해 **"몇 번 줄, 몇 번 프렛"** 인지 알아낸다.

> **[복붙 프롬프트]**
> "Phase 2의 H를 이용해 각 손끝(검지8·중지12·약지16·새끼20, 옵션 엄지4)의 이미지좌표를 지판좌표 `(u,v)`로 변환하는
> 함수 `fingerToFret(landmark)`를 만들어줘. **우쿨렐레 줄은 4개**다. 규칙:
> - **string(줄)** = `4 - round(v)` 로 매핑(v=0 → 4번줄(G) … v=3 → 1번줄(A)). v가 -0.4~3.4 밖이면 '지판 밖'.
> - **fret(프렛)** = 손끝이 프렛선 사이 어디에 있는지로 결정. 프렛 공간 n(프렛선 n-1과 n 사이)을 누르면 소리나는 프렛은 n.
>   평균율 보정 격자를 쓰면 u를 프렛선 경계 배열과 비교해 구간 판정. (단, 0~0.x 영역은 개방현 0)
> - 손끝이 실제로 줄을 '누르고 있는지'는 1차 버전에선 'u,v가 격자 안 + 프렛 공간 중앙 부근'으로 근사.
> - 결과를 화면에 디버그 표시: 각 손끝 옆에 `'2번줄 1프렛'`처럼 라벨."

**완료 기준:** 손가락을 지판 위에서 움직이면 라벨이 실제 줄·프렛과 일치하게 바뀐다.
첫 검증은 **C(1번줄 3프렛)·Am(4번줄 2프렛)** 단일 손가락으로 — 한 손가락만 보면 되므로 매핑 오류를 바로 잡을 수 있다.
**함정:** 우쿨렐레는 프렛이 좁아 경계 근처 떨림이 기타보다 심함 → Phase 5의 스무딩으로 해결. 일단은 raw로 정확도만 확인.

### ▶ Phase 4 — 목표 코드와 비교 + 판정 로직
**목표:** 현재 운지를 목표 코드와 대조해 **맞음/틀림 + 무엇이 틀렸는지**를 계산한다.

> **[복붙 프롬프트]**
> "2장의 우쿨렐레 `CHORDS` 데이터 모델을 추가하고, 코드 선택 드롭다운(C·Am·F·G·A·D·Dm·C7·G7·Em)을 만들어줘.
> `evaluateVoicing(detectedFingers, targetChord, options)` 함수 구현:
> - target에서 눌러야 할 음(fret>0)들의 집합 `required = [{string,fret,finger}]`.
> - detected에서 각 손끝의 `{finger, string, fret}`.
> - 각 required 음마다 매칭 결과 분류:
>   - ✅ **correct**: 같은 줄·같은 프렛에 손가락이 있음 (options.strictFinger면 손가락 번호까지 일치)
>   - ⚠️ **wrong_position**: 손가락은 짚었는데 줄/프렛이 어긋남(어느 방향으로 몇 칸인지 delta 계산)
>   - ❌ **missing**: 눌러야 할 음에 아무 손가락도 없음
>   - ➕ **extra**: 누르면 안 되는 곳을 누름
> - 전체 결과로 `score`(맞은 음/전체), `isCorrect`(임계 이상), 그리고 **교정 지시 리스트** 반환:
>   예) `{finger:3, from:{string:1,fret:2}, to:{string:1,fret:3}, msg:'약지를 브리지 쪽으로 한 프렛 (1번줄 3프렛)'}`
> - 난이도 옵션: `strictFinger`(손가락 번호 검사), `fretTolerance`, `requireMute`(우쿨렐레는 기본 false).
> ⚠️ C·Am·C7처럼 required가 1개뿐인 코드는 그 1음만 맞으면 isCorrect=true가 되도록 score 계산을 처리."

**완료 기준:** 코드 선택 후 손으로 짚으면 각 음이 ✅/⚠️/❌로 분류되고, 점수가 나온다.
C(1번줄 3프렛)·Am(4번줄 2프렛)을 정확히 잡으면 '정확합니다'가 떠야 한다.
**함정:** 손가락-음 매칭을 "가장 가까운 음에 그리디"로 하면 오매칭 가능 → 필요 시 헝가리안/최소비용 매칭. 1차는 그리디로 충분.

### ▶ Phase 5 — 교정 피드백 UX + 안정화(스무딩/홀드 게이트)
**목표:** "틀렸다"를 넘어 **어떻게 고칠지**를 시각·텍스트·음성으로 안내하고, 떨림을 잡는다.

> **[복붙 프롬프트]**
> "피드백과 안정화를 추가해줘.
> **시각 오버레이:**
> - 각 손끝 색상: 정답=초록, 위치오류=노랑, 불필요=빨강. 짚어야 하는데 빈 음은 지판 위에 점선 ◯ 타깃 표시.
> - 위치오류 손가락엔 현재 위치→목표 위치로 **화살표**를 그려 '이동 방향' 시각화.
> **텍스트 코칭 패널:** 교정 지시를 우선순위순으로 한국어 문장 리스트로.
>   예: '② 중지를 4번줄 2프렛으로', '검지를 2번줄 1프렛으로'. 전부 맞으면 '✅ 정확합니다!'.
>   ⚠️ 줄 방향은 **줄 번호로만** 표현(예: '1번줄 쪽으로', '4번줄 쪽으로'). '굵은/얇은 줄'이라는 말은 우쿨렐레 튜닝상 혼란을 주니 쓰지 마.
> **음성(옵션):** SpeechSynthesis로 가장 중요한 교정 1개를 읽어줌(너무 자주 말하지 않게 쿨다운).
> **안정화:**
> - 각 손끝 좌표에 **One-Euro 필터**(또는 이동평균)로 떨림 제거. (우쿨렐레는 프렛이 좁아 스무딩 강도를 기타보다 조금 세게)
> - **홀드 게이트**: 같은 판정이 N프레임(예: 0.7초) 연속 유지될 때만 '정답' 확정 → 깜빡임 방지.
> - 정답을 일정시간 유지하면 성공 효과(체크·사운드)."

**완료 기준:** 손을 살짝 떨어도 판정이 안정적. 틀린 손가락에 화살표가 뜨고, 그대로 옮기면 초록으로 바뀜.
**함정:** 음성이 매 프레임 떠들면 괴로움 → 메시지 변경 시에만, 쿨다운 두고 발화.

## 4. 핵심 알고리즘 메모 (AI가 헷갈릴 때 같이 붙여주기)

### 4-1. Homography 4점 (이미지→지판)
- 입력: 이미지 픽셀 4점 `src`, 대응 지판좌표 4점 `dst`(우쿨렐레: `(0,0),(0,3),(K,3),(K,0)`).
- 8×8 선형계를 풀어 3×3 행렬 H(마지막 원소=1) 산출(DLT). 변환: `[u,v,w]ᵀ = H·[x,y,1]ᵀ`, `(u/w, v/w)`.
- 역변환(지판→이미지)은 H⁻¹. 격자선 그릴 때 사용.

### 4-2. 프렛 위치(평균율, 비선형 보정)
- 너트~브리지 길이를 1로 볼 때, **프렛 n의 위치 = 1 − 2^(−n/12)**.
- 캘리브레이션이 너트~프렛K 구간만 덮으면, 이 비율을 K로 정규화해 내부 프렛선 배치:
  `ratio(n) = (1 - 2^(-n/12)) / (1 - 2^(-K/12))`, `u = ratio(n) * K`.

### 4-3. 손끝→줄/프렛 판정 (우쿨렐레 4줄)
- `(u,v) = H·fingertip`. `string = 4 - round(v)`(범위 검사 -0.4~3.4), 프렛은 u를 프렛선 경계 배열과 비교해 구간 결정.
- '실제로 눌렀는지'는 1차 근사(격자 안 + 프렛공간 중앙). 정밀화는 손끝의 깊이/굽힘으로 추정(Phase 6).

### 4-4. 손가락 매칭 (detected ↔ required)
- 1차: 각 required 음에 대해 가장 가까운(거리 최소) 손끝을 그리디 배정.
- 충돌·오배정 잦으면 비용행렬 만들어 최소비용 매칭으로 교체.
- 우쿨렐레는 required가 1~3개로 적어 그리디로도 충분히 안정적.

## 5. 테스트 시나리오 (각 Phase 검증용)

1. **빈손** → 손 안 잡힘/격자만 표시, 에러 없이 유지.
2. **C 코드 정확히(1번줄 3프렛, 약지)** → 1음 초록, '정확합니다'. (단일 손가락 → 가장 쉬운 첫 검증)
3. **Am 코드 정확히(4번줄 2프렛, 중지)** → 1음 초록. (줄 번호 매핑이 4번줄까지 맞는지 확인)
4. **C에서 약지를 2프렛에** → 노랑 + '약지를 1번줄 3프렛으로 (브리지 쪽 한 프렛)' + 화살표.
5. **F 코드(4번줄 2프렛 + 2번줄 1프렛)** → 2음 모두 초록. (2손가락 매칭 검증)
6. **G 코드(3·2·1번줄)** → 3음 분류 + 점수. (3손가락 매칭·줄 구분 검증)
7. **카메라 흔들기** → 재보정 안내, 오작동 시 복구.
8. **저조도/역광** → 인식률 저하 시 안내문.

- [ ] **카메라/악기 고정**: 움직이면 재보정 필요 — 버튼 상시 노출.
- [ ] **프렛 비선형성 + 좁은 넥**: 평균율 보정 또는 다중 클릭. 우쿨렐레는 더 촘촘.
- [ ] **판정 떨림**: 스무딩 + 홀드 게이트 없으면 깜빡여서 못 씀.
- [ ] **줄 호칭**: '굵은/얇은 줄' 금지, **N번줄**로만 안내.
- [ ] **단일 손가락 코드**: C·Am·C7은 required 1개 — score/isCorrect가 1음 기준으로 맞게 동작하는가?
- [ ] **음성 과다**: 쿨다운·변경 시에만 발화.
- [ ] **개인정보 안내**: 영상은 로컬에서만 처리되고 서버 전송 없음을 명시.

## 부록 B — 실제 바이브코딩용 "완성형 프롬프트 팩"

아래 프롬프트들은 Cursor, Claude Code, Codex, Copilot Chat 같은 AI 코딩 도구에 그대로 붙여 넣는 용도입니다.
처음부터 한 번에 모든 기능을 만들라고 시키면 실패 확률이 높으므로, **B-0 → B-1 → B-2... 순서대로 한 단계씩 실행**하세요.

### B-0. 프로젝트 킥오프 프롬프트

```text
너는 시니어 프론트엔드 엔지니어이자 컴퓨터비전 기반 인터랙티브 웹앱 개발자야.

우리는 "웹캠 손 제스처로 우쿨렐레 코드를 제대로 운지했는지 확인하고, 틀렸을 때 어느 손가락을 어느 줄/프렛으로 옮겨야 하는지 교정해주는 웹앱"을 만들 거야.

악기 전제 (매우 중요):
- 우쿨렐레는 줄이 4개다. 표준 튜닝 G·C·E·A.
- 줄 번호와 위치: 4번줄(G)=맨 위(연주자 쪽), 3번줄(C), 2번줄(E), 1번줄(A)=맨 아래.
- re-entrant 튜닝(4번 G가 high-G)이라 두께 순서는 기타와 다르다. 그래서 "굵은 줄/얇은 줄" 표현은 쓰지 말고 항상 "N번줄"로 부른다.
- 지판 좌표계: v=0 -> 4번줄, v=3 -> 1번줄. string = 4 - Math.round(v).
- 프렛 간격이 좁고 넥이 짧아 손끝 떨림/오판정이 기타보다 잘 난다. 스무딩과 홀드 게이트가 더 중요하다.

핵심 목표:
1. 웹캠 영상을 브라우저에서 로컬로 처리한다.
2. MediaPipe Hand Landmarker로 손가락 21개 랜드마크를 실시간 추적한다.
3. 사용자가 지판 4점을 클릭해 수동 캘리브레이션한다(줄 4개 기준).
4. homography로 손끝 좌표를 지판 좌표계로 변환한다.
5. 손끝 위치를 줄/프렛으로 판정한다.
6. 선택한 목표 우쿨렐레 코드와 비교해 정답/오답을 판정한다.
7. 틀린 운지에 대해 한국어 텍스트, 캔버스 화살표, 선택적 음성 피드백으로 교정한다.

MVP에서 절대 하지 말 것:
- 서버로 영상을 업로드하지 말 것.
- 처음부터 자동 지판 인식을 구현하지 말 것.
- 머신러닝 모델을 새로 학습하지 말 것.
- 완벽한 음향 기반 코드 판정을 섞지 말 것.
- 기타용 6줄 좌표계/코드 보이싱을 절대 재사용하지 말 것.

기술 스택:
- 기본은 바닐라 HTML/CSS/JavaScript.
- 필요하면 Vite를 사용해도 되지만, 상태관리 프레임워크는 도입하지 않는다.
- 손 추적은 @mediapipe/tasks-vision HandLandmarker를 사용한다.
- 오버레이는 Canvas 2D를 사용한다.
- 좌표 변환은 직접 구현한 homography 유틸을 사용한다.
- 음성 피드백은 Web Speech API SpeechSynthesis를 사용한다.

중요한 UX 원칙:
- 각 단계가 눈으로 검증 가능해야 한다.
- 카메라, 캘리브레이션, 손 추적, 판정, 피드백 상태를 화면에 명확히 표시한다.
- 사용자가 카메라/악기를 움직이면 재보정을 쉽게 할 수 있어야 한다.
- 판정이 흔들리지 않도록 스무딩과 홀드 게이트를 구현한다.
- 개인정보 안내: 영상은 브라우저 로컬에서만 처리된다고 명시한다.

이제 내가 Phase 단위로 요청할 것이다. 매 Phase마다:
1. 필요한 파일 변경을 먼저 요약하고,
2. 구현하고,
3. 직접 실행/검증 방법을 알려주고,
4. 다음 Phase로 넘어가기 전 확인해야 할 체크리스트를 출력해줘.

먼저 현재 프로젝트 구조를 확인하고, Phase 0을 진행할 준비가 됐다고 말해줘.
```

### B-3. Phase 2 상세 프롬프트 — 지판 수동 캘리브레이션 (줄 4개)

```text
Phase 2를 구현해줘. 목표는 사용자가 지판 4점을 클릭해 이미지 좌표를 우쿨렐레 지판 좌표계로 변환하는 homography를 만들고, 녹색 지판 격자를 그리는 것이다. 우쿨렐레는 줄이 4개(G·C·E·A)다.

캘리브레이션 UX:
1. "지판 보정" 버튼을 추가한다.
2. 버튼 클릭 시 calibrationMode = true.
3. 사용자가 다음 순서로 4점을 클릭하게 한다.
   - 1점: 4번줄(G) 쪽 너트 모서리
   - 2점: 1번줄(A) 쪽 너트 모서리
   - 3점: 1번줄(A) 쪽 K번 프렛 모서리
   - 4점: 4번줄(G) 쪽 K번 프렛 모서리
4. K 입력값을 제공한다. 기본값 5, 최소 3, 최대 12.
5. 클릭할 때마다 점과 번호를 canvas에 표시한다.
6. 현재 클릭해야 할 위치를 큰 안내 문구로 표시한다.
7. 4점이 모두 찍히면 homography를 계산하고 localStorage에 저장한다.
8. "보정 초기화" 버튼을 제공한다.

좌표계 정의:
- 이미지 픽셀 srcPts:
  - 사용자가 클릭한 4점
- 지판 dstPts (줄 4개라 v는 0~3):
  - p1 -> (0, 0)   // 4번줄, 너트
  - p2 -> (0, 3)   // 1번줄, 너트
  - p3 -> (K, 3)   // 1번줄, K프렛
  - p4 -> (K, 0)   // 4번줄, K프렛
- u축: 너트에서 브리지 방향, 프렛 진행 방향
- v축: 4번줄에서 1번줄 방향
- v=0은 4번줄(G), v=3은 1번줄(A)

필수 함수:
- computeHomography(srcPts, dstPts): 3x3 행렬 반환
- invert3x3(matrix): 3x3 역행렬 반환
- applyHomography(matrix, point): { x, y } 또는 { u, v } 반환
- solveLinearSystem(A, b): Gaussian elimination으로 8x8 풀이
- getFretLinePositions(K): 평균율 기반 프렛선 위치 배열 반환
  - ratio(n) = (1 - Math.pow(2, -n / 12)) / (1 - Math.pow(2, -K / 12))
  - n=0은 0, n=K는 1

격자 그리기:
- drawFretboardGrid(ctx)
- H inverse를 사용해 지판좌표 -> 이미지좌표로 투영한다.
- 프렛선:
  - n=0..K
  - 평균율 ratio를 u=ratio*K처럼 변환하거나 내부 헬퍼로 처리한다.
- 줄선:
  - v=0..3 (4줄)
  - 각 줄은 u=0..K를 잇는 선.
- 실제 줄 번호 라벨:
  - v=0: 4번줄(G)
  - v=1: 3번줄(C)
  - v=2: 2번줄(E)
  - v=3: 1번줄(A)
- 격자가 보정된 상태면 녹색, 보정 중이면 노란색으로 표시한다.

저장:
- localStorage key: ukuleleFingering.calibration.v1
- 저장 데이터:
  - clickedPoints
  - fretCountK
  - homographyImageToFretboard
  - homographyFretboardToImage
  - createdAt

완료 기준:
- 4점을 클릭하면 실제 지판 위에 녹색 격자(줄 4개·프렛선 K+1개)가 겹친다.
- 새로고침 후에도 보정값이 복원된다.
- 보정 초기화가 동작한다.

주의:
- canvas 클릭 좌표는 CSS 좌표가 아니라 canvas 내부 픽셀 좌표로 변환해야 한다.
- getBoundingClientRect()와 canvas.width/height 비율을 이용한다.
- 비디오/canvas 미러링이 적용된 경우 클릭 좌표도 같은 좌표계로 일관되게 처리한다.
- 우쿨렐레는 작으니, 클릭 직전 해당 영역을 확대해 보여주는 돋보기(zoom lens) 옵션을 추가하면 정밀도가 올라간다(선택).
```

### B-4. Phase 3 상세 프롬프트 — 손끝을 줄/프렛으로 변환

```text
Phase 3을 구현해줘. 목표는 MediaPipe 손끝 좌표를 지판 homography에 통과시켜 "몇 번 손가락이 몇 번 줄 몇 프렛에 있는지" 판정하는 것이다. 우쿨렐레는 줄이 4개다.

대상 손가락:
- 검지: finger 1, landmark 8
- 중지: finger 2, landmark 12
- 약지: finger 3, landmark 16
- 새끼: finger 4, landmark 20
- 엄지: T, landmark 4는 옵션으로만 표시한다.

필수 데이터 구조:
detectedFingers = [
  {
    finger: 1,
    name: "검지",
    landmarkIndex: 8,
    pixel: { x, y },
    fretboard: { u, v },
    string: 2,
    fret: 1,
    inFretboard: true,
    confidence: 0.87
  }
]

필수 함수:
1. getFingertipPixel(landmark)
   - MediaPipe normalized 좌표를 canvas pixel 좌표로 변환한다.
   - 미러링 정책을 Phase 1/2와 일관되게 적용한다(appState.isMirrored).
2. mapPixelToFretboard(pixel)
   - H imageToFretboard 적용.
   - { u, v } 반환.
3. getStringFromV(v)
   - v가 -0.4~3.4 밖이면 null.   // 우쿨렐레 4줄
   - nearestIndex = Math.round(v)
   - string = 4 - nearestIndex
   - 예: v=0 -> 4번줄(G), v=3 -> 1번줄(A)
4. getFretFromU(u, fretCountK)
   - u가 0보다 작거나 K보다 크면 null.
   - 평균율 기반 프렛 경계 배열을 사용한다.
   - u가 0 근처면 fret=0 가능.
   - 프렛 n은 프렛선 n-1과 n 사이를 누를 때 n으로 판정한다.
   - 예: 너트와 1프렛선 사이면 fret=1.
5. detectFingerPositions(handResults)
   - 현재 프레임의 손가락 위치 배열 반환.

디버그 표시:
- 각 손끝 옆에 "검지: 2번줄 1프렛" 라벨 표시.
- 지판 밖이면 "검지: 지판 밖".
- 오른쪽 패널에 detectedFingers JSON 요약을 사람이 읽기 쉽게 표시.
- 프렛/줄 판정이 흔들리는지 보기 위해 raw u,v 값도 소수점 2자리로 표시한다.

완료 기준:
- 손가락을 실제 1번줄 3프렛(C 코드 위치)에 두면 "1번줄 3프렛"으로 표시된다.
- 4번줄 2프렛(Am 위치), 2번줄 1프렛(F 위치) 등 초보 코드 위치에서 대체로 맞게 표시된다.
- 지판 밖에 손을 두면 지판 밖으로 표시된다.

주의:
- 우쿨렐레는 프렛이 좁아 손끝이 줄/프렛 경계 사이에서 더 잘 떨린다. 이 Phase에서는 raw 표시까지 구현하고, 안정화는 Phase 5에서 한다.
- 아직 "누르고 있음"과 "공중에 떠 있음"은 구분하지 않는다. MVP에서는 손끝 위치만으로 판단한다.
```

### B-5. Phase 4 상세 프롬프트 — 우쿨렐레 코드 데이터와 판정 엔진

```text

- C   (0003)  A줄 3프렛
- Am  (2000)  G줄 2프렛
- F   (2010)  G줄 2프렛, E줄 1프렛
- G   (0232)  C줄 2프렛, E줄 3프렛, A줄 2프렛
- A   (2100)  G줄 2프렛, C줄 1프렛
- D   (2220)  G·C·E줄 2프렛
- Dm  (2210)  G줄 2프렛, C줄 2프렛, E줄 1프렛
- C7  (0001)  A줄 1프렛
- G7  (0212)  C줄 2프렛, E줄 1프렛, A줄 2프렛
- Em  (0432)  C줄 4프렛, E줄 3프렛, A줄 2프렛

데이터 형식(배열은 4->1번줄 순서로 통일):
const CHORDS = {
  C: {
    id: "C",
    name: "C 메이저",
    strings: [
      { string: 4, fret: 0, finger: 0 },
      { string: 3, fret: 0, finger: 0 },
      { string: 2, fret: 0, finger: 0 },
      { string: 1, fret: 3, finger: 3 }
    ]
  }
  // ... 나머지 코드도 같은 형식
}

UI:
- 코드 선택 드롭다운.
- 엄격 모드 토글:
  - OFF: 줄/프렛만 맞으면 정답.
  - ON: 지정 손가락 번호까지 맞아야 정답.
- 뮤트 검사 토글(우쿨렐레는 기본 OFF — 보통 4줄 다 침):
  - OFF: x 줄은 경고만.
  - ON: x 줄에 손가락이 있으면 extra로 판단.
- 현재 점수 표시: 예: 2/2, 100%.
- 최종 상태:
  - 정확합니다
  - 거의 맞아요
  - 교정이 필요해요

필수 함수:
1. getRequiredFingeredNotes(chord)
   - fret > 0인 항목만 반환.
2. getOpenStrings(chord)
   - fret === 0.
3. getMutedStrings(chord)
   - fret === "x".  (우쿨렐레 기본 코드엔 거의 없음)
4. evaluateVoicing(detectedFingers, chord, options)
   반환 예:
   {
     score: 0.5,
     isCorrect: false,
     requiredResults: [
       {
         target: { string: 1, fret: 3, finger: 3 },
         status: "wrong_position",
         matchedFinger: { finger: 3, string: 1, fret: 2 },
         delta: { string: 0, fret: 1 },
         message: "약지를 1번줄 3프렛으로 브리지 쪽 한 프렛 이동하세요."

     extras: [],
     corrections: []

매칭 규칙:
- strictFinger=true면 target.finger와 detected.finger가 같은 것부터 찾는다.
- strictFinger=false면 같은 string/fret에 있는 아무 손가락을 허용한다.
- 같은 손가락이 여러 target에 매칭되지 않도록 used set을 둔다.
- wrong_position은 같은 finger가 감지되었지만 target 위치와 다른 경우.
- missing은 해당 target에 대응하는 손가락이 없거나 지판 밖인 경우.
- extra는 어떤 target에도 속하지 않는 감지 손가락.
- required가 1개뿐인 코드(C, Am, C7)는 그 1음만 맞으면 score=1, isCorrect=true가 되게 한다.

교정 메시지 생성:
- 손가락 이름:

  - 2 중지
  - 3 약지
  - 4 새끼
- 프렛 방향:
  - 현재 fret < 목표 fret: "브리지 쪽으로 N프렛"
  - 현재 fret > 목표 fret: "너트 쪽으로 N프렛"
- 줄 방향 (※ "굵은/얇은 줄" 금지, 줄 번호로만):
  - 현재 string > 목표 string: "4번줄 쪽으로 N줄"   (예: 2번 -> 4번 방향)
  - 현재 string < 목표 string: "1번줄 쪽으로 N줄"   (예: 3번 -> 1번 방향)
- 예:
  - "약지를 1번줄 3프렛으로 옮기세요."
  - "중지를 너트 쪽으로 1프렛 이동해 4번줄 2프렛에 놓으세요."
  - "검지는 2번줄 1프렛을 눌러야 합니다."

완료 기준:
- C 코드에서 1번줄 3프렛이 맞으면 정확하다고 나온다(단일 손가락).
- F 코드에서 4번줄 2프렛 + 2번줄 1프렛이 맞으면 정확하다고 나온다.
- 일부 손가락을 틀리게 두면 어떤 손가락을 어디로 옮길지 나온다.
- 손이 없으면 "손을 지판 위에 올려주세요" 상태가 나온다.

1. 각 손끝의 pixel 좌표와 변환된 fretboard 좌표 u,v를 화면에 표시한다.
2. v=0이 4번줄(G), v=3이 1번줄(A)로 매핑되는지 확인한다.
3. string = 4 - Math.round(v) 규칙이 맞는지 확인한다. (기타용 6 - round(v)가 남아있지 않은지)
4. u가 프렛선 경계 배열과 비교되어 fret으로 변환되는지 확인한다.
5. fret = Math.ceil(u) 같은 단순 계산을 쓰고 있다면 평균율 기반 경계 비교로 교체한다.
6. 프렛 경계 근처에서 떨리면 hysteresis 또는 moving average를 추가한다. (우쿨렐레는 프렛이 좁아 더 필요)
7. 사용자가 누르는 위치는 프렛 바로 뒤쪽이어야 하므로, 프렛 공간 중앙 기준 tolerance를 조절한다.

수정 후 디버그 패널에 raw u/v, filtered u/v, string/fret을 함께 표시해줘.
```

### C-4. 판정이 매 프레임 바뀌며 깜빡일 때

```text
코드 판정 결과가 매 프레임 바뀌어서 UX가 불안정하다.
스무딩과 홀드 게이트를 제대로 구현해줘.

요구사항:
- 각 손가락별 최근 6프레임 pixel 좌표 moving average. (우쿨렐레는 프렛이 좁아 기타보다 살짝 강하게)
- 같은 판정 signature가 300ms 이상 유지될 때 텍스트 코칭 갱신.
- correct 상태는 700ms 이상 유지될 때만 성공 확정.
- 음성 안내는 stableEvaluation 기준으로만 실행.
- 디버그 표시로 rawEvaluation과 stableEvaluation을 구분해서 보여줘.
```
